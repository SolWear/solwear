# Поток транзакции Solana (отправка SOL) — Solvare

Документ описывает сквозной сценарий отправки **SOL** с телефона при подписи на NFC-устройстве (часы) в архитектуре Solvare. Предполагается использование RPC Solana и протокола NDEF для передачи запроса подписи и получения подписи (см. `NFC_PROTOCOL_NDEF.md`).

---

## Сквозной поток (end-to-end)

### 1. Ввод данных на SendScreen

Пользователь на экране отправки вводит:

- адрес получателя (публичный ключ в Base58);  
- сумму в SOL (или в UI-единицах с последующей конвертацией в лампорты).

### 2. Валидация

- **Адрес:** проверка формата **Base58** и длины/декодирования в байты публичного ключа (32 байта для стандартного кошелька Solana).  
- **Сумма:** сравнение с балансом отправителя (с учётом резерва на комиссию); отклонение при недостаточных средствах или некорректном вводе.

### 3. Получение recent blockhash

Телефон вызывает RPC-метод **`getLatestBlockhash`** (с нужным `commitment`, например `confirmed`), получает `blockhash` для привязки транзакции ко времени и защиты от повторного воспроизведения.

### 4. Инструкция System Program — перевод

Строится **бинарная** инструкция программы **System Program**: перевод лампортов с аккаунта отправителя на аккаунт получателя. Формат соответствует спецификации Solana: индекс программы в списке аккаунтов, упакованные аккаунты, дискриминатор инструкции `Transfer`, поле `lamports` (little-endian u64).

### 5. Структура байтов транзакции (сообщение + место под подписи)

Сообщение транзакции Solana (transaction **message**) в бинарном виде включает:

1. **Заголовок сообщения** (`message header`): число подписей, число только для чтения, число только для чтения без подписи (в зависимости от версии сообщения — v0 с адресными таблицами или legacy).  
2. **Список ключей аккаунтов** (закодированный порядок: подписанты, затем прочие).  
3. **Recent blockhash** (32 байта).  
4. **Инструкции** — каждая: индекс программы, массив пар (индекс аккаунта, is_signer, is_writable в компактном виде в рамках кодировки сообщения), сырые `data` инструкции.

На этапе до подписи на часах телефон формирует **тело сообщения** и массив **заглушек подписей** (для одной подписи — 64 нулевых байта или согласованный с часами формат «unsigned»), чтобы итоговая структура **serialized transaction** имела правильный layout для подстановки одной реальной подписи Ed25519.

Итоговая передача на NFC в поле `tx_bytes` (Base64 в JSON `SIGN_REQUEST`) — **сериализованные байты**, согласованные с прошивкой часов (обычно полная legacy-транзакция или только `message` + метаданные — это должно быть зафиксировано в прошивке и в приложении одинаково).

### 6. Запись на NFC (SIGN_REQUEST)

Телефон записывает NDEF с типом `solvare:sign_request` и JSON, где `tx_bytes` — Base64 от сформированных байт (см. протокол NDEF).

### 7. Подпись на часах

Часы читают запрос, подписывают соответствующее сообщение **приватным ключом Ed25519**, записывают NDEF `solvare:sign_response` с `signature` в Base64 (64 байта подписи).

### 8. Чтение подписи и сборка транзакции

Телефон считывает `SIGN_RESPONSE`, декодирует Base64 → 64 байта подписи, **вставляет подпись** на место заглушки в структуре транзакции согласно порядку подписантов.

### 9. Отправка в сеть

Вызов RPC **`sendTransaction`** с закодированной в Base64 (или binary) готовой транзакцией и параметрами (`encoding`, `skipPreflight` по политике приложения).

### 10. Подтверждение

- Опрос **`getSignatureStatuses`** с интервалом **2 секунды**, не более **20 попыток** (таймаут ожидания подтверждения **~40 секунд**).  
- При успехе — состояние UI «подтверждено»; при истечении попыток или ошибке статуса — показ ошибки.

### 11. Итоговое состояние UI

Отображение **«подтверждено»** (signature `confirmed`/`finalized`) или **ошибка** с человекочитаемым текстом и при возможности кодом/логом для поддержки.

## Бинарный формат транзакции (кратко)

- **Legacy transaction:** сжатый массив подписей (каждая 64 байта) + закодированное **message**.  
- **Message:** header + account keys + blockhash + instructions.  
- Порядок аккаунтов и флаги writable/signer влияют на корректность подписи; часы должны подписывать **то же сообщение**, что ожидает сеть после подстановки подписи.

Подробная спецификация: [Solana Transaction Format](https://docs.solana.com/developing/programming-model/transactions) (официальная документация).

---

## Лампорты и комиссия

- **1 SOL = 1 000 000 000 лампортов** (10⁹).  
- Все суммы в инструкции `Transfer` задаются в **лампортах** (целое u64).  
- **Комиссия за транзакцию** удерживается сетью; для типичной простой транзакции с **одной подписью** порядок величины — **около 5000 лампортов** (точное значение зависит от текущих `fee` в сети и лимитов; приложение может запрашивать актуальные значения через RPC при необходимости).  
- При проверке баланса нужно оставить запас на комиссию, чтобы не получить ошибку «insufficient funds» после вычета fee.

## Сценарии ошибок

| Ситуация | Проявление | Примечание |
|----------|------------|------------|
| Неверный адрес | Ошибка валидации Base58 / длины | До RPC и NFC |
| Недостаточно средств | Ошибка от симуляции / `sendTransaction` / статуса | Учесть fee |
| Истёкший blockhash | Транзакция отклонена | Повторить с новым `getLatestBlockhash` |
| Сетевая ошибка | Таймаут RPC, 5xx, нет соединения | Повтор, офлайн-сообщение |
| Таймаут / сбой NFC | Нет ответа, ошибка чтения/записи NDEF | См. коды в `NFC_PROTOCOL_NDEF.md` |
| Ошибка проверки подписи | Сеть отклоняет транзакцию | Несовпадение message/порядка ключей с тем, что подписали часы |

---

## Связь с NFC-протоколом

- Шаги **6–8** напрямую соответствуют типам **`solvare:sign_request`** и **`solvare:sign_response`**.  
- Для согласованности реализации телефона и часов критично единообразно определять, что именно входит в `tx_bytes` (полная транзакция с пустой подписью или только message + явный хэш для подписи).
