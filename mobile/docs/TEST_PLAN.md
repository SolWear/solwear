# План тестирования Solvare

План покрывает модульные и интеграционные автотесты, ручные сценарии и негативные случаи NFC для Android-приложения Solvare (Solana NFC cold wallet).

---

## 1. Модульные тесты (Unit)

| Область | Что проверять |
|---------|----------------|
| **Base58** | Кодирование, декодирование, валидация адресов/ключей (некорректные символы, длина, контрольная сумма при наличии в формате) |
| **NdefProtocol** | Разбор входящих NDEF-сообщений в доменные структуры; формирование исходящих сообщений; граничные длины и неполные записи |
| **SolanaRepository** | Парсинг ответов RPC (баланс, ошибки, пустые/неожиданные поля JSON) |
| **PriceRepository** | Логика кэширования цены (TTL, инвалидация, повторный запрос после истечения срока) |
| **WalletViewModel** | Переходы состояния UI: загрузка, успех, ошибка, отсутствие кошелька, обновление баланса |
| **Сборщик байтов транзакции** | Корректность сериализации инструкций, лимитов, подписи-заглушек в тестах; сравнение с эталонными векторами или известными fixtures |

---

## 2. Интеграционные тесты (Integration)

| Сценарий | Описание |
|----------|----------|
| **Solana RPC (devnet)** | Реальные вызовы к devnet: получение баланса по известному адресу, обработка ошибок сети/таймаута (по политике CI — с моком при нестабильной среде) |
| **CoinGecko API** | Парсинг ответа API цены; устойчивость к частичному JSON и кодам ошибок HTTP |
| **Полный сценарий отправки** | Поток «собрать транзакцию → обмен через NFC» с **замоканным** NFC-адаптером; проверка итогового запроса к сети или состояния ViewModel |

---

## 3. Ручные тест-кейсы

Для каждого кейса ниже указаны **предусловия**, **шаги** и **ожидаемый результат**.

### TC-01: Первый запуск — пустое состояние

| Поле | Содержание |
|------|------------|
| **Предусловия** | Чистая установка приложения; кошелёк ранее не сопоставлялся с NFC (или сброшены данные приложения). |
| **Шаги** | 1. Запустить приложение. 2. Дождаться отображения главного экрана. |
| **Ожидаемый результат** | Отображается пустое состояние / приглашение подключить кошелёк; баланс не показан или равен нулю согласно дизайну; нет аварийного завершения. |

### TC-02: Подключение кошелька через NFC

| Поле | Содержание |
|------|------------|
| **Предусловия** | Установлено приложение; NFC на телефоне включён; доступен тег с корректными данными кошелька Solvare. |
| **Шаги** | 1. На главном экране нажать обновление / действие «подключить через NFC» (как в UI). 2. Приложить NFC-тег к устройству. 3. Дождаться завершения чтения. |
| **Ожидаемый результат** | Кошелёк считается подключённым; отображается публичный адрес (если предусмотрено UI); показывается баланс SOL с devnet/mainnet согласно конфигурации. |

### TC-03: Обновление баланса

| Поле | Содержание |
|------|------------|
| **Предусловия** | Кошелёк подключён (TC-02). Сеть доступна. |
| **Шаги** | 1. Запомнить отображаемый баланс (или изменить баланс на стороне сети тестовым переводом). 2. Нажать кнопку обновления баланса. |
| **Ожидаемый результат** | Индикатор загрузки (если есть) завершается; баланс соответствует актуальным данным RPC; время/состояние кэша обновляется согласно логике приложения. |

### TC-04: Отображение эквивалента в USD

| Поле | Содержание |
|------|------------|
| **Предусловия** | Кошелёк подключён; цена SOL успешно получается (CoinGecko или мок в тестовой сборке). |
| **Шаги** | 1. Открыть главный экран с балансом. 2. Убедиться в наличии строки с USD. |
| **Ожидаемый результат** | Под или рядом с балансом SOL отображается эквивалент в USD; значение согласовано с курсом и суммой SOL (допустимы округления по спецификации UI). |

### TC-05: Переход на экран отправки

| Поле | Содержание |
|------|------------|
| **Предусловия** | Пользователь на главном экране; кошелёк подключён. |
| **Шаги** | 1. Нажать кнопку «Отправить». |
| **Ожидаемый результат** | Открывается экран (или фрагмент) отправки SOL с полями адреса и суммы; навигация «Назад» возвращает на главный экран. |

### TC-06: Валидация пустых полей

| Поле | Содержание |
|------|------------|
| **Предусловия** | Открыт экран отправки. |
| **Шаги** | 1. Оставить поля адреса и/или суммы пустыми. 2. Нажать отправку / далее. |
| **Ожидаемый результат** | Отображаются сообщения об ошибках валидации (пустой адрес, пустая сумма); транзакция не формируется; NFC-подпись не запрашивается. |

### TC-07: Некорректный Solana-адрес

| Поле | Содержание |
|------|------------|
| **Предусловия** | Открыт экран отправки. |
| **Шаги** | 1. Ввести заведомо неверную строку (не Base58-адрес, слишком короткая/длинная). 2. Подтвердить отправку. |
| **Ожидаемый результат** | Показывается сообщение вида **«Некорректный Solana-адрес»** (или эквивент по глоссарию приложения); переход к NFC не выполняется. |

### TC-08: Недостаточно средств

| Поле | Содержание |
|------|------------|
| **Предусловия** | Известен баланс кошелька; сумма отправки больше доступного с учётом комиссии. |
| **Шаги** | 1. Ввести валидный адрес получателя. 2. Ввести сумму, превышающую доступный баланс. 3. Инициировать отправку. |
| **Ожидаемый результат** | Отображается ошибка о недостатке средств; запрос подписи через NFC не выполняется (или отменяется на этапе валидации). |

### TC-09: Отправка SOL (полный поток с NFC)

| Поле | Содержание |
|------|------------|
| **Предусловия** | Кошелёк подключён; достаточный баланс; часы/тег готовы к подписи; сеть стабильна. |
| **Шаги** | 1. Заполнить корректный адрес и сумму. 2. Подтвердить отправку. 3. Выполнить подпись через NFC согласно подсказкам UI. 4. Дождаться подтверждения в приложении и при необходимости в обозревателе. |
| **Ожидаемый результат** | Транзакция принята сетью; UI показывает успех / хэш / ссылку на обозреватель; баланс после обновления уменьшился на сумму и комиссию. |

### TC-10: NFC недоступен

| Поле | Содержание |
|------|------------|
| **Предусловия** | NFC отключён в настройках системы **или** устройство без NFC. |
| **Шаги** | 1. Попытаться подключить кошелёк или отправить SOL с шагом, требующим NFC. |
| **Ожидаемый результат** | Приложение не падает; показывается понятное сообщение о необходимости включить NFC или о неподдерживаемом устройстве; без бесконечного спиннера. |

### TC-11: Таймаут NFC

| Поле | Содержание |
|------|------------|
| **Предусловия** | Кошелёк подключён; сценарий отправки доведён до шага ожидания NFC; тег **не** прикладывается. |
| **Шаги** | 1. Запустить ожидание NFC. 2. Дождаться истечения таймаута. |
| **Ожидаемый результат** | В bottom sheet (или аналогичном UI) отображается ошибка таймаута; можно закрыть и повторить попытку; состояние формы сохраняется по спецификации. |

### TC-12: Ошибка сети при загрузке баланса

| Поле | Содержание |
|------|------------|
| **Предусловия** | Включён режим «без сети» или заблокирован RPC (firewall / неверный endpoint в debug). |
| **Шаги** | 1. Открыть приложение или нажать обновление баланса. |
| **Ожидаемый результат** | Отображается сообщение об ошибке сети; показывается последний известный кэшированный баланс **или** явное пустое состояние — согласно реализованной политике; повторная попытка возможна. |

### TC-13: Ошибка сети при отправке

| Поле | Содержание |
|------|------------|
| **Предусловия** | Транзакция подписана через NFC; на этапе отправки в RPC сеть недоступна. |
| **Шаги** | 1. Выполнить подпись. 2. Симулировать обрыв сети до получения ответа от RPC. |
| **Ожидаемый результат** | Пользователь видит ошибку отправки; при повторном появлении сети можно повторить отправку **или** приложение различает «неизвестный результат» и не дублирует перевод без проверки (зафиксировать фактическое поведение в отчёте). |

### TC-14: Таймаут подтверждения транзакции

| Поле | Содержание |
|------|------------|
| **Предусловия** | Отправка инициирована; сеть сильно замедлена или подтверждение имитируется таймаутом в тестовой среде. |
| **Шаги** | 1. Отправить транзакцию. 2. Дождаться истечения таймаута ожидания подтверждения в UI. |
| **Ожидаемый результат** | Показывается сообщение о таймауте подтверждения; предлагается проверить обозреватель по signature; нет молчаливого зависания. |

## 4. Негативные сценарии NFC

| Сценарий | Предусловия | Шаги | Ожидаемый результат |
|----------|-------------|------|---------------------|
| **Тег убран во время чтения** | Начато чтение NDEF. | Убрать тег до завершения. | Ошибка чтения; UI позволяет повторить; нет краша; при частичных данных — отказ с понятным текстом. |
| **Тег убран во время записи** | Начата запись/обмен для подписи. | Убрать тег в процессе. | Транзакция не считается успешной без полной подписи; сообщение об ошибке; повтор возможен. |
| **Неверный тип тега** | Пользователь приложил обычный NDEF URL или пустой тег. | Выполнить действие, ожидающее кошелёк Solvare. | Сообщение о несовместимом теге; без записи в некорректный формат. |
| **Повреждённые / усечённые данные** | Тег с битой полезной нагрузкой или обрезанной записью. | Прочитать тег. | Парсер отклоняет данные; ошибка валидации протокола; без использования как валидного ключа/tx. |
| **Чужой формат NDEF** | Корректный NDEF, но не протокол Solvare. | Подключение или подпись. | Явный отказ; не смешивать с успешным подключением кошелька. |

---

## 5. Критерии приёмки тестирования

- Все перечисленные **unit**-области покрыты тестами или явно исключены с обоснованием в PR.
- Интеграционные тесты RPC/API стабильны в CI (при необходимости — с моками и отдельной меткой `devnet`).
- Ручные кейсы **TC-01–TC-14** и негативные NFC пройдены на релиз-кандидате перед выкладкой.
- Регрессия: повтор TC-02, TC-09 после значимых изменений NFC или сборки транзакций.

*Документ следует обновлять при добавлении экранов, смене строк UI или протокола NFC.*
