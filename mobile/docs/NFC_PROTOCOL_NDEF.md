# Спецификация протокола NFC NDEF — Solvare

## Обзор

Коммуникация между телефоном (Android) и часами (носителем ключей) в проекте Solvare построена на **NDEF** (NFC Data Exchange Format). Сообщения передаются в виде записей NDEF с внешним типом (`TNF_EXTERNAL_TYPE`), идентифицируемым пространством имён Solvare. Полезная нагрузка — JSON в кодировке UTF-8.

Протокол рассчитан на сценарий «холодного» кошелька: приватный ключ остаётся на часах; телефон только читает публичный ключ и передаёт на подпись сериализованные байты транзакции.

**Версия протокола:** `1`.

---

## Типы сообщений

Все записи используют **TNF = External** (`0x04`). Поле **Type** — строка в формате `namespace:type` (в терминах Android NDEF — внешний тип с доменом приложения).

| Тип записи (Type)     | Назначение |
|------------------------|------------|
| `solvare:wallet`       | Публичный ключ кошелька и сеть |
| `solvare:sign_request` | Запрос подписи транзакции |
| `solvare:sign_response`| Ответ с подписью |

### READ_WALLET (`solvare:wallet`)

- **TNF:** External  
- **Type:** `solvare:wallet`  
- **Payload (JSON):**

```json
{
  "version": 1,
  "pubkey": "<base58>",
  "network": "devnet"
}
```

- `version` — версия полезной нагрузки (совпадает с версией протокола для MVP).  
- `pubkey` — публичный ключ Solana в кодировке Base58.  
- `network` — идентификатор сети (например, `devnet`, `mainnet-beta`).

### SIGN_REQUEST (`solvare:sign_request`)

- **TNF:** External  
- **Type:** `solvare:sign_request`  
- **Payload (JSON):**

```json
{
  "version": 1,
  "tx_bytes": "<base64>"
}
```

- `tx_bytes` — сериализованное тело сообщения транзакции Solana (или полная структура, согласованная с прошивкой часов) в Base64.

### SIGN_RESPONSE (`solvare:sign_response`)

- **TNF:** External  
- **Type:** `solvare:sign_response`  
- **Payload (JSON):**

```json
{
  "version": 1,
  "signature": "<base64>"
}
```

- `signature` — 64-байтовая подпись Ed25519 (или эквивалент в соглашении Solana) в Base64.

## Поток чтения кошелька

1. Пользователь подносит устройство с NDEF-меткой (часы) к телефону.  
2. Телефон обнаруживает тег и считывает NDEF-сообщение (через foreground dispatch или аналогичный механизм).  
3. Парсер находит запись с типом `solvare:wallet`.  
4. Декодируется JSON; проверяется `version`.  
5. Из поля `pubkey` извлекается публичный ключ; при необходимости используется `network` для выбора RPC.

Итог: телефон получает адрес кошелька без передачи приватного ключа.

---

## Поток подписи (MVP, два касания)

Для MVP предполагается **два последовательных взаимодействия** с меткой (два «тапа»):

1. **Шаг 1 — запись:** телефон записывает на тег NDEF-сообщение с записью `solvare:sign_request` (JSON с `tx_bytes`).  
2. **Шаг 2 — часы:** прошивка часов читает запрос, подписывает байты приватным ключом, перезаписывает/обновляет NDEF ответом `solvare:sign_response`.  
3. **Шаг 3 — чтение:** пользователь снова подносит часы; телефон читает `solvare:sign_response`, извлекает `signature` и продолжает сборку/отправку транзакции.

Таким образом: **запись запроса → (обработка на часах) → чтение ответа**.

## Конечный автомат NFC (состояния)

Рекомендуемая модель состояний на стороне телефона:

| Состояние            | Описание |
|---------------------|----------|
| `IDLE`              | Ожидание действия пользователя, NFC не активен для сценария или готов к новому циклу. |
| `WAITING_FOR_TAG`   | Пользователь инициировал операцию; ожидается обнаружение метки. |
| `READING`           | Выполняется чтение NDEF с метки. |
| `WRITING`           | Выполняется запись NDEF на метку. |
| `SUCCESS`           | Операция завершена успешно; можно вернуться в `IDLE`. |
| `ERROR`             | Ошибка (таймаут, неверный формат, отказ тега и т.д.); показ сообщения и переход в `IDLE` или повтор. |

Типичные переходы: `IDLE` → `WAITING_FOR_TAG` → `READING` / `WRITING` → `SUCCESS` | `ERROR` → `IDLE`.

## Foreground dispatch

На Android для стабильной работы с NFC в активном экране приложения рекомендуется:

- Регистрировать **foreground dispatch** в `onResume` и снимать в `onPause` (или использовать современные API в духе `NfcAdapter` + `enableForegroundDispatch` / `ReaderMode` в зависимости от minSdk).  
- Указывать **Intent filters** и/или **tech lists** так, чтобы приложение получало намерения при обнаружении NDEF-совместимых тегов, пока активно соответствующее `Activity`.  
- Обрабатывать `ACTION_NDEF_DISCOVERED` (и при необходимости `ACTION_TAG_DISCOVERED`) для извлечения `Parcelable` тега и чтения/записи NDEF.

Это снижает вероятность перехвата события другим приложением и даёт предсказуемое поведение при «тапе».

## Коды ошибок и обработка

Рекомендуется единообразно классифицировать сбои на уровне приложения (логика поверх системных исключений):

| Код / категория        | Причина | Действие |
|------------------------|---------|----------|
| `ERR_TAG_NOT_FOUND`    | Метка не обнаружена вовремя | Подсказка пользователю, повтор |
| `ERR_NDEF_PARSE`       | Нет нужной записи или битый JSON | Сообщение о несовместимой метке |
| `ERR_VERSION_MISMATCH` | `version` в JSON ≠ поддерживаемой | Запрос обновления прошивки/приложения |
| `ERR_WRITE_FAILED`     | Запись NDEF отклонена (защита, ёмкость) | Повтор, проверка прав доступа к тегу |
| `ERR_READ_FAILED`      | Чтение не удалось | Повтор касания |
| `ERR_TYPE_UNKNOWN`     | Type не из набора Solvare | Игнор или ошибка в зависимости от режима |

Системные коды Android (`IOException`, коды от `Tag`) следует маппить на эти категории для UI и логирования.

## Таймауты

Явный таймаут протокола на уровне приложения для одного касания **не задаётся отдельно**: действует **системный таймаут NFC-сеанса** чтения/записи (зависит от устройства и версии ОС). Приложение должно:

- не блокировать UI дольше допустимого;  
- при отсутствии ответа переводить автомат в `ERROR` и предлагать повтор.

Для полного цикла подписи в MVP учитываются **два касания**; общее время определяется действиями пользователя и системой.

## Ограничения NDEF по сравнению с APDU / ISO-DEP

| Аспект | NDEF | APDU (ISO-DEP) |
|--------|------|----------------|
| Модель данных | Сообщения/записи, часто одна «страница» | Поток команд-ответов, произвольная длина сессии |
| Размер полезной нагрузки | Ограничен ёмкостью NDEF на теге (часто килобайты) | Зависит от чипа и протокола, обычно гибче |
| Латентность | Запись + отдельное чтение (MVP) | Можно несколько обменов за одну сессию |
| Стандартизация для кошельков | Условная, доменные типы приложения | Распространены смарт-карточные профили |
| Безопасность | Как у носимого NDEF-тега; важна прошивка часов | Можно ввести выделенный апплет, защищённые каналы |

NDEF подходит для **MVP** и простых сценариев «одна запись — один ответ»; для больших транзакций, сессионной безопасности и интерактивного протокола предпочтительнее **APDU поверх ISO-DEP** (Host Card Emulation или физический NFC-чип в часах).

## Путь миграции на APDU / ISO-DEP (будущие версии)

1. **Версионирование:** ввести поле `transport` или поднять `version` протокола с сохранением обратной совместимости там, где возможно.  
2. **Селекция AID / приложения:** определить AID для апплета Solvare на часах.  
3. **Формат команд:** замапить текущие смыслы `sign_request` / `sign_response` на APDU (например, `INS_SIGN`, передача длины и фрагментов `tx_bytes`).  
4. **Совместимость:** приложение на телефоне определяет возможности тега (NDEF vs ISO-DEP) и выбирает стек.  
5. **Документация:** отдельный документ `NFC_PROTOCOL_APDU.md` с таблицей INS/CLA и кодами статуса SW1/SW2.

До миграции все реализации должны соблюдать данную NDEF-спецификацию версии **1**.
