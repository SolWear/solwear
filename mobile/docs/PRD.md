# PRD: Solvare — Android-приложение «холодный» кошелёк Solana через NFC

**Версия документа:** 1.0  
**Продукт:** Solvare  
**Платформа:** Android  
**Статус:** MVP

---

## 1. Обзор и цели

Solvare — мобильное приложение для Android, позволяющее держателям SOL использовать **холодное хранение** с подписью транзакций на **умных часах** (или совместимом NFC-устройстве), а телефон выступает как интерфейс просмотра баланса и инициирования переводов с обменом данными по **NFC (NDEF)**.

**Цель MVP:** безопасный минимальный цикл «просмотр баланса → обновление по NFC → отправка SOL с подписью по NFC» на **devnet**, только **нативный SOL**, без SPL и без истории операций.

---

## 2. Область (Scope) — MVP

### 2.1 Входит в MVP

- Главный экран с отображением баланса SOL и ориентировочной стоимости в USD.
- Обновление данных кошелька через NFC (чтение/запись NDEF по сценарию приложения).
- Отправка SOL: ввод адреса и суммы, валидация, подпись через NFC, отправка в сеть, подтверждение результата.
- Интеграция с **Solana RPC** (devnet).
- Получение курса SOL/USD (например, **CoinGecko**).
- Обработка ошибок для основных состояний (сеть, NFC, валидация, RPC).
- Требование **NFC** на устройстве; **minSdk 24**.

### 2.2 Не входит в MVP (Out of scope)

- Список токенов (SPL) и балансы токенов.
- История транзакций и детальный эксплорер внутри приложения.
- Несколько аккаунтов / несколько кошельков в одном приложении.
- Версия для **iOS**.
- Mainnet / продакшен-сеть (см. нефункциональные требования).

---

## 3. Целевая аудитория

**Пользователи:** держатели криптовалюты, которые хотят **холодное хранение** и готовы использовать **умные часы** (или другое устройство с NFC) как компонент, отвечающий за подпись или обмен ключевыми данными, не подключая полноценный «горячий» кошелёк на телефоне к той же модели угроз.

**Jobs-to-be-done (MVP):**

- Быстро увидеть баланс SOL и эквивалент в USD.
- Актуализировать данные без ручного ввода секретов на экране телефона (через NFC к часам).
- Отправить SOL, не раскрывая приватный ключ на телефоне — подпись через NFC.

---

## 4. Пользовательские истории и критерии приёмки

### US-1: Просмотр баланса SOL на главном экране

**Как** держатель SOL, **я хочу** видеть свой баланс крупно и ориентир в USD, **чтобы** быстро оценить состояние кошелька.

**Критерии приёмки:**

1. На главном экране отображается **крупное** значение баланса в SOL (формат с разумным числом знаков после запятой, согласованный с UI-спекой).
2. Под или рядом с балансом SOL отображается **меньшим** шрифтом и приглушённым стилем **приблизительный** эквивалент в USD на основе актуального курса.
3. Если курс временно недоступен, USD-блок показывает понятное состояние (например, «—» или «Курс недоступен») без падения экрана.
4. Если кошелёк ещё не «подключён» / нет данных с часов, отображается **empty state** (см. UI-спеку), а не нулевой баланс как факт.

---

### US-2: Обновление данных кошелька через NFC

**Как** пользователь, **я хочу** обновить баланс и связанные данные, поднеся часы к телефону, **чтобы** данные соответствовали состоянию на устройстве-хранилище.

**Критерии приёмки:**

1. На главном экране есть явное действие **«Обновить»** (иконка), по нажатию открывается модальное окно / bottom sheet с инструкцией **поднести часы**.
2. В модалке отображаются состояния: ожидание NFC, чтение/запись, успех, ошибка с возможностью **повторить**.
3. После успешного обмена по NFC главный экран показывает **актуальный** баланс SOL и пересчитанный USD (при доступном курсе).
4. При ошибке NFC или таймауте пользователь получает **понятное** сообщение и может закрыть модалку или повторить попытку.
5. Обмен данными использует **NDEF read/write** в рамках протокола приложения (детали — в технической спецификации / ADR).

---

### US-3: Отправка SOL на другой кошелёк

**Как** пользователь, **я хочу** отправить SOL на адрес получателя, **чтобы** перевести средства, подписав транзакцию на часах через NFC.

**Критерии приёмки:**

1. С экрана отправки можно ввести **адрес получателя** и **сумму** в SOL.
2. Валидация выполняется **до** попытки NFC: неверный адрес / некорректная сумма / сумма больше баланса — показываются **inline**-ошибки.
3. Кнопка основного действия (например, «Подписать и отправить») запускает цепочку: подготовка транзакции → NFC для подписи → отправка в RPC → отображение статуса (см. UI-спеку).
4. После подтверждения в сети пользователь видит **идентификатор транзакции** (signature) в успешном состоянии.
5. При сбое на любом этапе (NFC, RPC, таймаут подтверждения) показывается **ошибка с текстом** и возможностью корректного выхода или повтора там, где это применимо.

## 5. Функциональные требования

| ID | Требование | Примечание |
|----|------------|------------|
| FR-1 | **NFC NDEF:** чтение и запись NDEF-сообщений для обмена данными с часами | Формат payload — согласовать с прошивкой/приложением часов |
| FR-2 | **Solana RPC:** запрос баланса, построение и отправка подписанной транзакции, ожидание подтверждения | Только devnet в MVP |
| FR-3 | **CoinGecko (или эквивалент):** курс SOL/USD для отображения ориентира | Кэширование и деградация при недоступности API |
| FR-4 | **Обработка ошибок:** сеть, парсинг, NFC недоступен / прерван, RPC-ошибки, пользователь отменил, превышен таймаут | Единые UX-паттерны (см. UI-спеку) |
| FR-5 | **Без SPL:** операции только с нативным SOL | Любые SPL-адреса/токены вне скоупа |
| FR-6 | **Один кошелёк в контексте сессии MVP** | Смена устройства/кошелька — только через повторный NFC-флоу |

---

## 6. Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-1 | **Сеть:** только **Solana Devnet** для MVP |
| NFR-2 | **Активы:** только **SOL**, без SPL |
| NFR-3 | **minSdkVersion:** **24** |
| NFR-4 | **NFC:** обязательное наличие на устройстве; при отсутствии NFC — блокирующее объяснение на старте или в манифесте/feature |
| NFR-5 | **Безопасность:** секретные ключи не отображаются в UI; минимизация логирования чувствительных данных |
| NFR-6 | **Производительность:** отзывчивый UI при ожидании RPC/NFC; таймауты с понятной обратной связью |

---

## 7. Зависимости и интеграции

- **Solana RPC** (devnet): баланс, sendTransaction, подтверждение.
- **CoinGecko API** (или согласованный аналог): цена SOL/USD.
- **NFC / NDEF:** контракт обмена с прошивкой или companion-приложением часов.

---

## 8. Предположения (Assumptions)

1. На часах реализован совместимый слой обмена данными через NFC (NDEF), согласованный с мобильным приложением.
2. Пользователь понимает, что USD — **ориентир по курсу**, а не котировка для исполнения сделок.
3. Devnet-достаточность для MVP: тестовые SOL, ожидаемые сбои и реорганизации цепочки приемлемы для обучения и демо.
4. Устройство пользователя соответствует **minSdk 24** и имеет рабочий NFC-чип и разрешения ОС.

---

## 9. Принятые решения (Decisions)

| Тема | Решение | Обоснование |
|------|---------|---------------|
| Сеть MVP | Devnet | Снижение риска и регуляторной нагрузки на первом этапе |
| Актив | Только SOL | Упрощение подписи, UI и тестирования |
| Подпись | Через NFC с часами | Соответствие продуктовой идее «cold via watch» |
| Курс | Внешний API (CoinGecko) | Быстрый ориентир без собственной инфраструктуры цен |

---

## 10. Открытые вопросы (Open questions)

1. Точный **бинарный/JSON формат** NDEF payload и версионирование протокола между Android и часами.
2. Нужна ли **явная индикация devnet** в UI постоянно (баннер), или достаточно настроек/онбординга.
3. Политика **таймаутов** NFC и RPC (числа секунд) и число автоматических повторов для отправки транзакции.
4. Поведение при **частичном** успехе (транзакция отправлена, но подтверждение не получено за таймаут) — как объяснять пользователю и предлагать проверку в эксплорере (вне приложения).
5. Локализация: только русский интерфейс в MVP или RU + EN с первого релиза.
